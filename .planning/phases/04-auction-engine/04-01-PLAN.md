**Prior decisions affecting this plan:**

- Money remains integer cents only, so bid increments, fee math, and settlement snapshots must avoid float persistence entirely.
- Listings already publish directly into active auction rows, so Phase 4 should extend that path instead of inventing a separate activation state machine.
- The current production topology is a single long-lived app instance, which makes a process-level overdue-auction sweep acceptable for this phase while still keeping close logic idempotent and row-lock safe.

**Concerns this plan must cover:**

- Concurrent bids and buyouts must resolve through a locked interactive transaction, not the current fast-query path.
- Close logic has to be reusable across on-demand mutation checks and the background sweep so winners, cancellations, and settlement rows cannot drift.
</context>
