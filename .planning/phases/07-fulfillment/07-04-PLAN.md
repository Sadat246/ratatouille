<objective>
Build the fulfillment service layer — the single set of server-only mutation and read entry points that Plans 07-05, 07-06, 07-07, and 07-08 will consume. Five tightly-scoped modules:

1. `errors.ts` — discriminated error class so route handlers can convert service-layer failures into typed JSON without sniffing `error.message`.
2. `idempotency.ts` — Uber webhook dedup helpers (mirrors `lib/payments/idempotency.ts` pattern but writes to `uber_direct_webhook_events`).
3. `notify.ts` — `notifyFulfillmentStatusChange(id, newStatus)` stub Phase 8 will replace; called from every status transition site so Phase 8 only needs to fill in the body.
4. `queries.ts` — three read helpers: `getFulfillmentForConsumer`, `getOpenFulfillmentsForBusiness`, `getRecentlyCompletedFulfillmentsForBusiness`. All enforce ownership.
5. `service.ts` — four mutation entry points: `choosePickup`, `chooseDelivery`, `verifyPickupCode`, `applyUberStatusUpdate`. Each performs the state-machine guard inside an interactive Postgres transaction; Uber Direct external calls happen OUTSIDE the transaction (Phase 6 invariant).

Purpose: Plans 07-05/06/07/08 become very thin — they only do auth + parse + call into this service layer + format response. The state-machine rules from RESEARCH § Fulfillment State Machine and the locked decisions D-04..D-14 are encoded once, here, and tested via the route handlers + downstream UI.

   * Codes returned by the fulfillment service layer. Route handlers map
   * these to HTTP statuses + JSON bodies; UI components surface the

   * Single source of truth — adding a new failure mode means adding ONE

  export type FulfillmentErrorCode =
    | "FULFILLMENT_NOT_FOUND"        // wrong id, or buyer/business mismatch (collapse to 404)
    | "FULFILLMENT_STATE_INVALID"    // tried to choose-pickup on already-decided row

    readonly code: FulfillmentErrorCode;
    readonly userMessage: string;

    constructor(code: FulfillmentErrorCode, userMessage: string, cause?: unknown) {
      super(`${code}: ${userMessage}`);
      this.code = code;
      this.userMessage = userMessage;
      if (cause !== undefined) {

      }

   * call this from their route handlers.

      case "FULFILLMENT_NOT_FOUND":
        return 404;
      case "FULFILLMENT_STATE_INVALID":
      case "PICKUP_CODE_INVALID":
      case "VALIDATION_FAILED":
      case "RECIPIENT_PHONE_INVALID":
      case "DELIVERY_ADDRESS_MISSING":
      case "BUSINESS_ADDRESS_MISSING":

      case "PICKUP_CODE_EXPIRED":
        return 410;
      case "PICKUP_CODE_GENERATION_FAILED":
      case "DELIVERY_PROVIDER_FAILED":

  ```typescript
  import "server-only";

  import { db } from "@/db/client";
  import { uberDirectWebhookEvents } from "@/db/schema";

  /**
   * Has this Uber Direct webhook event already been written to the
   * idempotency log? Plan 07-06's webhook route checks this BEFORE
   * applying the status update so retried deliveries (Uber retries up
   * to 4 times with exponential backoff) do not double-write status.
   */
  export async function wasUberEventProcessed(
    eventId: string,
  ): Promise<boolean> {
    const row = await db.query.uberDirectWebhookEvents.findFirst({
      columns: { eventId: true },
      where: (table, operators) => operators.eq(table.eventId, eventId),
    });
    return Boolean(row);
  }

  /**
   * Record this event id as processed. Uses `onConflictDoNothing` against
   * the unique index on `event_id` so a concurrent retry handler losing
   * the race becomes a clean no-op (no exception thrown).
   */
  export async function markUberEventProcessed(
    eventId: string,
    eventType: string,
  ): Promise<void> {
    await db
      .insert(uberDirectWebhookEvents)
      .values({
        eventId,
        eventType,
        processedAt: new Date(),
      })
      .onConflictDoNothing({ target: uberDirectWebhookEvents.eventId });
  }

  /**
   * Some Uber webhook payloads do not include a stable `id`; fall back
   * to a synthetic `delivery_id + status + timestamp_bucket` composite
   * so retries within the same status still dedup. The `created` epoch
   * is bucketed to 5 seconds — coarser than Uber's retry interval (10s
   * minimum) but tight enough that two distinct status changes are not
   * conflated.
   */
  export function deriveUberEventId(payload: {
    id?: string | null;
    delivery_id?: string | null;
    status?: string | null;
    created?: string | number | null;
  }): string {
    if (payload.id && payload.id.length > 0) {
      return payload.id;
    }
    const ts = typeof payload.created === "number"
      ? payload.created
      : payload.created
        ? Date.parse(payload.created)
        : Date.now();
    const bucket = Math.floor(ts / 5_000);
    return `derived:${payload.delivery_id ?? "unknown"}:${payload.status ?? "unknown"}:${bucket}`;
  }
  ```

  Notes:
  - `deriveUberEventId` is exported so Plan 07-06's webhook route always has *some* string to pass into `wasUberEventProcessed`. Per RESEARCH § Webhook payload, `id` is documented but variable across event versions; the fallback keeps the dedup contract intact.
  - The 5-second bucket is generous (Uber's first retry is 10s) but tight enough that two distinct events with the same status never conflate — Uber sends each new status as its own event.

    businessId: string,
    statuses: string[],
    limit: number,

    const rows = await db
      .select({
        id: fulfillments.id,
        settlementId: fulfillments.settlementId,
        mode: fulfillments.mode,
        status: fulfillments.status,
        deliveryProvider: fulfillments.deliveryProvider,
        pickupCode: fulfillments.pickupCode,
        pickupCodeExpiresAt: fulfillments.pickupCodeExpiresAt,
        deliveryReferenceId: fulfillments.deliveryReferenceId,
        trackingUrl: fulfillments.trackingUrl,
        deliveredAt: fulfillments.deliveredAt,
        createdAt: fulfillments.createdAt,
        updatedAt: fulfillments.updatedAt,
        listingId: listings.id,
        listingTitle: listings.title,
        buyerDisplayName: users.name,
      })
      .from(fulfillments)
      .innerJoin(settlements, eq(settlements.id, fulfillments.settlementId))
      .innerJoin(listings, eq(listings.id, fulfillments.listingId))
      .leftJoin(users, eq(users.id, settlements.buyerUserId))
      .where(
        and(
          eq(settlements.businessId, businessId),
          inArray(fulfillments.status, statuses),
        ),
      )
      .orderBy(desc(fulfillments.updatedAt))
      .limit(limit);

    return rows.map((row) => ({
      id: row.id,
      settlementId: row.settlementId,
      mode: row.mode as "pickup" | "delivery",
      status: row.status,
      deliveryProvider: row.deliveryProvider as "none" | "uber_direct",

      deliveredAt: row.deliveredAt,
      createdAt: row.createdAt,
      updatedAt: row.updatedAt,

  async function lockFulfillment(

  export async function choosePickup(input: {
    fulfillmentId: string;
    consumerUserId: string;
  }): Promise<ChoosePickupResult> {
    // Pickup-code generation is a (potentially-multi-shot) read against
    // the live DB; doing it BEFORE the transaction avoids holding a row
    // lock while waiting on the uniqueness probe.
    const pickupCode = await generateUniquePickupCode({ db }).catch((err) => {
      throw new FulfillmentServiceError(
        "PICKUP_CODE_GENERATION_FAILED",
        "Could not issue a pickup code; please retry in a moment.",
        err,
      );
    });
    const expiresAt = new Date(Date.now() + PICKUP_CODE_TTL_MS);

    const result = await getInteractiveDb().transaction(async (tx) => {
      const row = await lockFulfillment(tx, input.fulfillmentId);
      if (!row) {
        throw new FulfillmentServiceError(
          "FULFILLMENT_NOT_FOUND",
          "We could not find this order.",
        );
      }

      const settlement = await tx.query.settlements.findFirst({
        where: eq(settlements.id, row.settlementId),
        columns: { buyerUserId: true },
      });
      if (!settlement || settlement.buyerUserId !== input.consumerUserId) {
        throw new FulfillmentServiceError(
          "FULFILLMENT_NOT_FOUND",
          "We could not find this order.",
        );
      }

      if (row.status !== "pending_choice") {
        throw new FulfillmentServiceError(
          "FULFILLMENT_STATE_INVALID",
          "Fulfillment choice has already been made for this order.",
        );
      }

      const now = new Date();
      await tx
        .update(fulfillments)
        .set({
          mode: "pickup",
          status: "awaiting_business",
          deliveryProvider: "none",
          pickupCode,
          pickupCodeExpiresAt: expiresAt,
          updatedAt: now,
        })
        .where(eq(fulfillments.id, row.id));

      return {
        fulfillmentId: row.id,
        settlementId: row.settlementId,
        pickupCode,
        pickupCodeExpiresAt: expiresAt,
      } satisfies ChoosePickupResult;
    });

  export async function chooseDelivery(input: {

            addressLine2: true,
            city: true,
            state: true,

  // -- Webhook applies Uber status update ---------------------------------

  export type ApplyUberStatusUpdateResult = {
    fulfillmentId: string | null;
    appliedStatus: string | null;
    skipped: boolean;
    reason?: "unknown_status" | "no_matching_fulfillment" | "already_terminal";
  };

  /**
   * Applied by the Uber Direct webhook route (Plan 07-06) after signature
   * verification + idempotency dedup. Looks up the fulfillment by
   * `delivery_reference_id`, projects the Uber status to a schema status,
   * and writes idempotently. Caller is expected to have already
   * registered the event id via `markUberEventProcessed`.
   */
  export async function applyUberStatusUpdate(input: {
    deliveryReferenceId: string;
    uberStatus: string;
  }): Promise<ApplyUberStatusUpdateResult> {
    const projected = mapUberStatusToFulfillmentStatus(input.uberStatus);
    if (!projected) {
      return {
        fulfillmentId: null,
        appliedStatus: null,
        skipped: true,
        reason: "unknown_status",
      };
    }

    const result = await getInteractiveDb().transaction(async (tx) => {
      const found = await tx.execute(sql<LockRow>`
        select
          id,
          settlement_id as "settlementId",
          listing_id    as "listingId",
          mode,
          status,
          pickup_code   as "pickupCode",
          pickup_code_expires_at as "pickupCodeExpiresAt"
        from fulfillments
        where delivery_reference_id = ${input.deliveryReferenceId}
        for update
      `);
      const row = (found.rows[0] as LockRow | undefined) ?? null;
      if (!row) {
        return {
          fulfillmentId: null,
          appliedStatus: null,
          skipped: true,
          reason: "no_matching_fulfillment" as const,
        };
      }

      // Pitfall §6 — idempotent against terminal states.
      if (row.status === "delivered" || row.status === "failed" || row.status === "cancelled") {
        return {
          fulfillmentId: row.id,
          appliedStatus: row.status,
          skipped: true,
          reason: "already_terminal" as const,
        };
      }

      const now = new Date();
      const isDeliveredEvent = UBER_TERMINAL_DELIVERED.has(input.uberStatus as UberDeliveryStatus);

      await tx
        .update(fulfillments)
        .set({
          status: projected,
          deliveredAt: isDeliveredEvent ? now : undefined,
          updatedAt: now,
        })
        .where(eq(fulfillments.id, row.id));

      return {
        fulfillmentId: row.id,
        appliedStatus: projected,
        skipped: false,
      };
    });

    if (
      !result.skipped &&
      result.fulfillmentId &&
      UBER_NOTIFY_CONSUMER.has(input.uberStatus as UberDeliveryStatus)
    ) {
      const notifyStatus =
        projected === "delivered"
          ? "delivered"
          : projected === "failed"
            ? "failed"
            : "out_for_delivery";
      await notifyFulfillmentStatusChange(result.fulfillmentId, notifyStatus);
    }

    return result;
  }

  Implementation notes:
  - `lockFulfillment` is a thin helper that takes the tx and runs the raw `for update` SELECT. Same pattern Phase 6 uses for `settlements`.
  - **Two-phase commit pattern in `chooseDelivery`** (transaction → external API → second non-tx update) MIRRORS the Phase 6 invariant: external I/O never inside a Drizzle transaction. The `delivery_requested` status is written before the Uber call so a crash mid-call leaves the row inspectable. The follow-up update inserts `deliveryReferenceId` + `trackingUrl` once Uber responds.
  - `verifyPickupCode` does NOT re-check business membership — Plan 07-06's route handler does that with a `business_memberships` query before invoking the service. This is the same separation as Phase 6 (`authorizeApiRole` is at the route layer; service trusts ownership).
