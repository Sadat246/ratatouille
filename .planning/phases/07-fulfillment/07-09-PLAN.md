  ## Uber Direct (local development)

  Phase 7 fulfillment uses Uber Direct for the delivery branch (decision D-01).
  Four env vars are read in `lib/fulfillment/uber-direct.ts`:

  - `UBER_DIRECT_CLIENT_ID` — OAuth 2 client id from your Uber Direct app.
  - `UBER_DIRECT_CLIENT_SECRET` — OAuth 2 client secret.
  - `UBER_DIRECT_CUSTOMER_ID` — sandbox customer/business id Uber assigns
    to the developer account.
  - `UBER_DIRECT_WEBHOOK_SECRET` — shared secret Uber signs each webhook
    payload with (HMAC-SHA256). Used by `lib/fulfillment/uber-signature.ts`.

  Sign up for a sandbox account at https://developer.uber.com/docs/deliveries
  and create a Direct application; the four values appear in the dashboard.

  **Stub mode (the demo default).** If any of the four env vars is missing,
  `UBER_DIRECT_ENABLED` is `false` and `createUberDelivery` returns a stub
  payload (`deliveryReferenceId` starting with `stub_`, `trackingUrl` of
  `https://stub.uber.local/track/<id>`, `stub: true`). The consumer order
  page surfaces a "Demo mode" badge so reviewers know no real courier was
  dispatched. This keeps the entire delivery branch reviewable without an
  Uber account — you can still drive status transitions by hand-firing a
  signed webhook (recipe in the demo walk-through below).

  When real credentials are configured, the same code path uses the
  `uber-direct` SDK (https://www.npmjs.com/package/uber-direct) and the
  webhook receiver at `/api/webhooks/uber-direct` accepts genuine signed
  payloads forwarded from Uber's servers. Production-style local testing
  will need a public tunnel (`ngrok http 3000` or similar) registered as
  the sandbox webhook URL.

  ### Phase 7: Fulfillment (pickup + delivery)

  Picks up where the Phase 6 happy path leaves off: a captured settlement
  produces a `fulfillments` row in `pending_choice`, and the consumer
  decides between in-store pickup (a 6-digit code, 48-hour expiry) and

  #### C. Hand-firing a signed Uber webhook (no Uber account needed)

  Useful in stub mode to walk the row through `out_for_delivery` and
  `delivered` so the consumer + seller pages show the terminal states.

  Set `UBER_DIRECT_WEBHOOK_SECRET` to any non-empty value in `.env.local`
  (the verifier just needs the same secret on both sides — for local
  hand-firing, anything works):

  ```bash
  UBER_DIRECT_WEBHOOK_SECRET=local_dev_secret_change_me
  ```

  Then, with the dev server running, send a `pickup` event followed by a
  `delivered` event. Replace `del_xxx` with the value of
  `delivery_reference_id` you observed in step B.3:

  ```bash
  SECRET=local_dev_secret_change_me
  DELIVERY_ID=del_xxx

  # 1. Courier picked up the order at the store -> status flips to
  #    out_for_delivery on the consumer + seller pages.
  BODY=$(printf '{"id":"evt_local_1","kind":"event.delivery_status","delivery_id":"%s","status":"pickup","created":"2025-12-01T00:00:00Z"}' "$DELIVERY_ID")
  SIG=$(printf "%s" "$BODY" | openssl dgst -sha256 -hmac "$SECRET" -hex | awk '{print $2}')
  curl -i -X POST \
    -H "x-uber-signature: $SIG" \
    -H "content-type: application/json" \
    -d "$BODY" \
    http://localhost:3000/api/webhooks/uber-direct

  # 2. Courier delivered. status -> delivered, deliveredAt = now.
  BODY=$(printf '{"id":"evt_local_2","kind":"event.delivery_status","delivery_id":"%s","status":"delivered","created":"2025-12-01T00:30:00Z"}' "$DELIVERY_ID")
  SIG=$(printf "%s" "$BODY" | openssl dgst -sha256 -hmac "$SECRET" -hex | awk '{print $2}')
  curl -i -X POST \
    -H "x-uber-signature: $SIG" \
    -H "content-type: application/json" \
    -d "$BODY" \
    http://localhost:3000/api/webhooks/uber-direct
  ```

  Both calls return `{"received":true,"applied":true}`. Re-firing the
  same event id returns `{"received":true,"deduped":true}` (idempotency
  table `uber_direct_webhook_events`). Refresh the consumer's
  `/order/<settlementId>` page — it now shows the "Delivered" card.
  Refresh the seller's `/sell/fulfillments` page — the row moves to
  "Recently completed" with badge "Delivered".

  #### What to verify in the database

  - Every fulfillment row begins life in `pending_choice` (created by the
    Phase 6 `payment_intent.succeeded` webhook handler).
  - A successful pickup choice writes `mode='pickup'`,
    `status='awaiting_business'`, `pickup_code` (6 digits), and
    `pickup_code_expires_at = now() + 48h`.
  - A successful delivery choice writes `mode='delivery'`,
    `status='delivery_requested'`, `delivery_provider='uber_direct'`,
    `delivery_reference_id`, and `tracking_url`.
  - `verifyPickupCode` mutates only `status='picked_up'` +
    `delivered_at = now()`; the original `pickup_code` stays in the row
    as audit history.
  - Replayed webhooks land in `uber_direct_webhook_events` (unique on
    `event_id`); the second insert is `onConflictDoNothing` and the
    handler short-circuits with `deduped: true`.
  ```

  Notes:
  - The `curl` recipe uses `openssl dgst -sha256 -hmac` because it ships with macOS by default and is one-line on Linux. Reviewers without `openssl` can substitute their preferred HMAC tool — the format is documented enough to reproduce.
  - The recipe deliberately uses two events (pickup → delivered) rather than the full chain (`pickup` → `dropoff` → `delivered`) because the status-map projects them all to the same state changes and a 2-step demo is easier to follow.
  - "What to verify in the database" section parallels the Phase 6 "What to verify on the Stripe Dashboard" subsection — same audit-trail mindset, different surface.
