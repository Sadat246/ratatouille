<objective>
Wire the two consumer-facing fulfillment-choice API routes that Plan 07-07's order page submits to. Both routes are intentionally THIN — auth check via `authorizeApiRole("consumer")`, parse params/body, call the matching `service.ts` entry, project the result to the UI-friendly response shape, and convert any `FulfillmentServiceError` to the standard `{ ok: false, error: { code, message } }` envelope. No business logic lives in either route file.

Purpose: D-04 puts the choice screen on the post-payment confirmation page; the page (Plan 07-07) needs two POST endpoints to drive the "Pickup" and "Delivery" cards. By keeping the routes thin and the service layer fat, every state-machine guard (`pending_choice` only, ownership check, etc.) is enforced in exactly one place even though both routes appear identical from a control-flow perspective.

Output: Two route handlers + one shared HTTP helper. All three files compile under `npx tsc --noEmit`. Manual smoke from `curl` (or the Plan 07-07 UI when it lands) returns the expected envelopes for happy and error paths.
</objective>

  ```typescript
  import { NextResponse } from "next/server";

  import {
    FulfillmentServiceError,
    fulfillmentErrorStatus,
  } from "@/lib/fulfillment/errors";

  export function jsonFulfillmentError(
    code: string,
    message: string,
    status: number,
  ) {
    return NextResponse.json(
      {
        ok: false,
        error: { code, message },
      },
      { status },
    );
  }

  export function toFulfillmentErrorResponse(error: unknown) {
    if (error instanceof FulfillmentServiceError) {
      return jsonFulfillmentError(
        error.code,
        error.userMessage,
        fulfillmentErrorStatus(error.code),
      );
    }

    console.error("fulfillment route failed", error);
    return jsonFulfillmentError(
      "FULFILLMENT_INTERNAL_ERROR",
      "The fulfillment request failed. Try again in a moment.",
      500,
    );
  }
  ```

  ```typescript
  import { NextResponse } from "next/server";

  import { authorizeApiRole } from "@/lib/auth/api";
  import { toFulfillmentErrorResponse } from "@/lib/fulfillment/http";
  import { formatPickupCode } from "@/lib/fulfillment/pickup-code";
  import { choosePickup } from "@/lib/fulfillment/service";

  export const runtime = "nodejs";

  export async function POST(
    _request: Request,
    context: { params: Promise<{ id: string }> },
  ) {
    const authorization = await authorizeApiRole("consumer");
    if (!authorization.ok) {
      return NextResponse.json(authorization.body, {
        status: authorization.status,
      });
    }

    try {
      const { id: fulfillmentId } = await context.params;
      const result = await choosePickup({
        fulfillmentId,
        consumerUserId: authorization.session.user.id,
      });

      return NextResponse.json({
        ok: true,
        fulfillmentId: result.fulfillmentId,
        settlementId: result.settlementId,
        pickupCode: result.pickupCode,
        pickupCodeFormatted: formatPickupCode(result.pickupCode),
        pickupCodeExpiresAt: result.pickupCodeExpiresAt.toISOString(),
      });
    } catch (error) {
      return toFulfillmentErrorResponse(error);
    }
  }
  ```

  Notes:

  - Both `pickupCode` (raw 6 digits, for staff entry) and `pickupCodeFormatted` (display "482 619") are returned — UI can choose. Date is ISO-stringified so JSON serialisation is safe.

  ```typescript
  import { NextResponse } from "next/server";

  import { authorizeApiRole } from "@/lib/auth/api";
  import {
    jsonFulfillmentError,
    toFulfillmentErrorResponse,
  } from "@/lib/fulfillment/http";
  import { chooseDelivery } from "@/lib/fulfillment/service";

  export const runtime = "nodejs";

  type ChooseDeliveryBody = {
    recipientPhone?: unknown;
  };

  export async function POST(
    request: Request,
    context: { params: Promise<{ id: string }> },
  ) {
    const authorization = await authorizeApiRole("consumer");
    if (!authorization.ok) {
      return NextResponse.json(authorization.body, {
        status: authorization.status,
      });
    }

    let body: ChooseDeliveryBody;
    try {
      body = (await request.json()) as ChooseDeliveryBody;
    } catch {
      return jsonFulfillmentError(
        "VALIDATION_FAILED",
        "Request body must be JSON.",
        400,
      );
    }

    const recipientPhone =
      typeof body.recipientPhone === "string" ? body.recipientPhone : "";

    try {
      const { id: fulfillmentId } = await context.params;
      const result = await chooseDelivery({
        fulfillmentId,
        consumerUserId: authorization.session.user.id,
        recipientPhone,
      });

      return NextResponse.json({
        ok: true,
        fulfillmentId: result.fulfillmentId,
        settlementId: result.settlementId,
        deliveryReferenceId: result.deliveryReferenceId,
        trackingUrl: result.trackingUrl,
        stub: result.stub,
      });
    } catch (error) {
      return toFulfillmentErrorResponse(error);
    }
  }
  ```

  Notes:
  - The route narrows `body.recipientPhone` to `string | ""` and lets the service layer reject empty/short — keeping validation in one place. Service throws `RECIPIENT_PHONE_INVALID` (HTTP 422) for that case.
  - JSON-parse failure produces a 400 + `VALIDATION_FAILED` envelope (separate from the service-layer 422 because malformed JSON is a different class of failure than malformed field).
  - `stub: result.stub` is forwarded to the client so Plan 07-07's UI MAY surface a "Demo mode" badge if it chooses (not required for v1).
  - The `body.recipientPhone` `unknown` typing is intentional — explicit narrowing beats `any`.
